# CS-255
System Analysis and Design: SNHU 2026

The client for this project was DriverPass, a start-up company in the driver education industry. DriverPass identified a significant gap in the market: more than 65% of students fail their DMV driving tests because their preparation relies on memorizing old exams rather than receiving structured, current instruction. To solve this problem, Liam (the owner) and Ian (the IT officer) asked our consulting firm, Creative Technology Solutions, to design a secure, cloud-based web application that would allow students to take online practice tests, enroll in behind-the-wheel driving packages, and schedule lesson appointments from any device. The system also needed to support office secretaries managing appointments by phone, an IT officer managing all user accounts, and management generating downloadable activity reports.

I believe I handled the requirements gathering and translation into system design particularly well. After reviewing the interview transcript carefully, I was able to identify not just the stated features but also the implied constraints — things like the need to prevent double-booking, enforce role-based access control, and handle concurrent user sessions safely. I organized the business requirements document into clear functional and non-functional categories, which made the transition into the system design document much smoother. The security section was also a strength: I addressed authentication, encryption, brute-force protection, and payment compliance in a way that directly traced back to specific client concerns raised in the interview, rather than treating security as a generic checklist.

If I could revise one part, I would go deeper in the UML class diagram and its connection to the technical requirements. While the class diagram identifies the key entities — Customer, Reservation, Instructor, Vehicle, Package, and so on — I would improve it by adding method signatures to show how the classes interact at the code level, and by explicitly mapping each class to a database table in the technical requirements section. This would create a tighter link between the design documentation and the actual implementation work a development team would need to do, making the handoff from design to development cleaner and reducing the risk of misinterpretation.

I interpreted the user's needs by reading the interview transcript closely and listening for the problems behind the requests, not just the feature descriptions. For example, when Liam said he wanted to "see who made changes and when," I recognized that as a requirement for a complete, timestamped activity log — not just a basic record. I then carried each need forward from the business requirements document into the system design by tracing every use case to a class, every workflow to an activity diagram, and every security concern to a specific technical control. Considering user needs is essential because a technically flawless system that does not solve the right problem for the right people is ultimately a failure. 

My approach to software design starts with understanding before building. I use the interview or discovery phase to identify actors, goals, and constraints before writing a single requirement. From there, I move from high-level to low-level: use case diagrams to map actors and interactions, activity diagrams to trace the logic of key workflows, sequence diagrams to clarify how components communicate, and class diagrams to formalize the data model. In the future, I would continue to use this object-oriented, iterative approach, and I would add more explicit validation checkpoints — going back to the client after each major diagram to confirm that the design still matches their vision before moving forward. I would also incorporate user stories earlier in the process to keep the human experience at the center of every technical decision, and I would use a structured requirements traceability matrix to ensure that every requirement is accounted for in the final design with no gaps.
